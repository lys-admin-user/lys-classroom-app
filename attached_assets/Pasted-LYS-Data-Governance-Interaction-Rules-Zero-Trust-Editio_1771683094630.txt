LYS Data Governance & Interaction Rules: Zero-Trust EditionThis document defines the "Hard-Coded Laws" of the LYS platform. To ensure a 60-day secure global launch, these rules are enforced at the API and Database levels from Day 1.1. The "Success" Ledger: Immutable IntegrityThe "Success / Not Yet" metric is the platform's primary asset. It is treated like a financial ledger.Atomic Submissions: Once the Success Scanner submits a mark, the record is immediately FINALIZED and timestamped.The 24-Hour Edit Window: Teachers have exactly 1,440 minutes to correct a clerical error.Rule: At minute 1,441, the is_mutable flag in the DB toggles to FALSE.Override: Any change after this requires a signed "Audit Reason" which is permanently attached to the student's longitudinal record.Deletion Policy: No student "Success" data can be deleted. "Archiving" only hides data from the UI; the underlying data remains for the 10-year predictive engine.2. The "Day 1" Safety Intercept (Communication)To prevent delays in District approvals, we are implementing Automated Interception instead of manual review queues where possible.Real-Time PII Scrubbing: All outbound student messages are passed through a regex and NER (Named Entity Recognition) filter.Rule: If a student attempts to send a phone number, home address, or private social media handle, the message is blocked immediately with a "Safety Warning" notification to the student.Cross-Tenant Lockdown: Students cannot message users outside of their tenant_id (School/District) unless a "Global Project" flag is explicitly enabled by a Super Admin.The Archive Mandate: All metadata (IP, timestamp, device ID) for every communication is logged. Deleting a chat thread only removes it from the user's view; it remains in the Safety Vault for legal compliance.3. Mandatory Multi-Tenant Separation (RLS)The "Apartment Building" is enforced by the database engine, not the application code.Tenant Scoping: Every SQL query is automatically appended with WHERE tenant_id = current_user_tenant_id().Leakage Prevention: If a developer attempts a query without a tenant_id filter, the database will return an Access Denied error by default.4. Global Data Residency (Sovereignty)To comply with Nigeria (NDPR) and EU (GDPR) laws on Day 1:Regional Routing: The system uses Latent-Based Routing. Users with a .ng or .eu IP are automatically directed to isolated regional database clusters.Sync Logic: "Knowing" data (curriculum) is synced globally, but "Being" and "Doing" data (PII/Student work) is geofenced and never leaves the origin region.5. Automated COPPA ComplianceThe Age Gate: Upon account creation, if age < 13, the account is placed in a "Restricted State."Hard Rule: No messaging, no social sharing, and no public profile visibility until a verified Parental Consent Token is received via the Parent Portal.6. Author Marketplace SecurityMalicious Content Filter: All "Bricks" uploaded by Author Teachers are scanned for malicious scripts and "Hidden PII" before they are listed in the Spotify Library.Conflict Resolution: If a Teacher modifies a standard tag, it creates a Local Instance. The Global Library (The "Knowing" source of truth) remains read-only to prevent "Standard Drift."7. VPN & Fraud Protection3-Strike Rule: If GeoIP location and Payment Region mismatch for 3 consecutive sessions, the "Master Builder" AI is disabled until the user verifies their location. This prevents "Pricing Arbitrage" while maintaining access to basic portfolio tools.
